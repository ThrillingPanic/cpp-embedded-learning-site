Understanding Control Flow: Making Your Program Smart

Control flow is the beating heart of interactive programs. It's how your program makes decisions and reacts to what it senses. Without control flow, your program would be like a robot that always does the same thing—useful maybe three times, useless the other 999 times.

Imagine a thermostat without control flow: It would heat the house constantly, regardless of temperature. The house would become unbearably hot. The system would waste energy. It's useless because it can't decide "am I hot? If so, stop heating. Am I cold? If so, start heating."

Real embedded systems make thousands of decisions every second:

  • A medical monitor: "Is patient's heart rate normal? If abnormal, sound alarm"
  • A car's engine control: "Is oxygen sensor reading strange? If so, enter safe mode"
  • An industrial robot: "Did I reach the target position? If not, adjust motors"
  • A drone: "Is wind too strong? If so, return to launch site"

Control flow statements (if, else if, else, switch) are how embedded systems read sensors, make decisions, and take action. Master these concepts and you're well on your way to writing intelligent code.

The three fundamental control structures in embedded C++:

  • if: "Is this condition true? If YES, execute this block. If NO, skip it"
  • if...else: "Is this condition true? If YES, execute block A. If NO, execute block B" (choose one or the other)
  • switch: "What is this value? If 1, do this. If 2, do that. If 3, do something else" (pick one from many options)

The if Statement: "Do This Only If..."

The if statement is the simplest decision. It means: "Check this condition. If it's true, execute the code block. If it's false, skip it."

Think of it like a security checkpoint: "Is it raining? If YES, close the roof. If NO, ignore this." If it's not raining, you don't close the roof. Simple.

Syntax:

if (condition) {
    // Statements here execute ONLY if condition is true
    // If condition is false, this block is skipped completely
}

Real-World Embedded Examples

Example 1: Low Battery Protection

uint8_t battery_percent = 15;

if (battery_percent < 20) {
    // Battery is critically low
    enterLowPowerMode();  // Only runs if battery is below 20%
    displayBatteryWarning();
    disableWifi();  // Conserve power
}
// If battery_percent is 50, these functions never execute

Example 2: Temperature Safety

uint8_t motor_temperature = 85;

if (motor_temperature > 80) {
    // Motor is overheating—take action
    reduceMotorPower();
    activateCooling();
    logTemperatureEvent();
}
// If motor_temperature is 70, nothing happens—motor is fine

Example 3: Sensor Sanity Check

uint16_t pressure_reading = 985;

if (pressure_reading < 500) {
    // Pressure is impossibly low (sensor malfunction?)
    reportSensorError();
    useDefaultPressure();
}
// If pressure is reasonable (>500), code continues normally

Critical Mistake: Forgetting Braces

⚠️ Always Use Braces in if Statements

C++ allows single-line if statements without braces. This is a disaster waiting to happen:

// DANGEROUS - no braces
if (error)
    log_error();
    shutdown_system();  // This ALWAYS runs, regardless of error!

// The second line is NOT part of the if statement.
// It executes after the if regardless of condition.
// Very subtle bug!

// CORRECT - with braces
if (error) {
    log_error();
    shutdown_system();  // Now correctly tied to the if
}

Golden rule: Always use braces around if blocks, even for single statements. Your future self won't accidentally add bugs.

Nested if Statements: Multiple Conditions

Sometimes you need to make a decision inside a decision. This is called nesting:

// Device startup sequence
if (power_button_pressed) {
    // First decision: is the button being held?
    
    if (self_diagnostics_passed) {
        // Second decision: did self-test pass?
        
        if (battery_voltage_good) {
            // Third decision: is battery healthy?
            
            start_system();  // Only if ALL conditions are true
        } else {
            show_low_battery_error();
        }
    } else {
        show_self_test_failed_message();
    }
} else {
    // Power button not pressed—don't start
}

Nesting deepens logic complexity. Three levels is getting hard to follow. Five levels is a nightmare. More than three levels usually signals you should refactor into separate functions.

The if-else Statement

Execute one block if true, different block if false.

if (sensor_reading > THRESHOLD) {
    turn_on_motor();
} else {
    turn_off_motor();
}

if-else if-else Chain

uint8_t temperature = 35;

if (temperature > 40) {
    shutdown_device();
} else if (temperature > 35) {
    reduce_clock_speed();
} else if (temperature > 25) {
    normal_operation();
} else {
    increase_performance();
}

// Note: Only ONE block executes!

Nested if Statements

if (device_powered_on) {
    if (self_test_passed) {
        if (sensors_ready) {
            start_measurement();
        } else {
            wait_for_sensors();
        }
    } else {
        report_fault();
    }
} else {
    enter_sleep_mode();
}

Problem: Gets hard to read. Consider combining conditions instead:

// Clearer
if (device_powered_on && self_test_passed && sensors_ready) {
    start_measurement();
}

The switch Statement

Use when checking one variable against multiple values. Cleaner than many if-else chains.

uint8_t state = 2;

switch (state) {
    case 0:
        init_device();
        break;
    case 1:
        normal_operation();
        break;
    case 2:
        error_detected();
        break;
    default:
        unknown_state();
        break;
}

The Break Statement (Critical!)

// WRONG - No break statements!
switch (mode) {
    case 1:
        set_high_speed();    // Runs...
    case 2:
        set_low_speed();     // ...then this runs too!
    case 3:
        set_standby();       // ...and this!
}

// RIGHT - break prevents fall-through
switch (mode) {
    case 1:
        set_high_speed();
        break;
    case 2:
        set_low_speed();
        break;
    case 3:
        set_standby();
        break;
    default:
        invalid_mode();
        break;
}

Intentional Fall-through

Sometimes you want code to execute for multiple cases:

switch (error_code) {
    case 1:          // Fall through
    case 2:
    case 3:
        // Handle errors 1, 2, 3 the same way
        log_mild_error();
        continue_operation();
        break;
    case 4:          // Different handling
    case 5:
        // Handle errors 4, 5 differently
        shutdown_device();
        break;
}

Practical Embedded Example

// UART command parser
uint8_t command = read_uart();

switch (command) {
    case 'S':  // Start
        start_measurement();
        uart_send("Started\n");
        break;
    case 'P':  // Pause
        pause_measurement();
        uart_send("Paused\n");
        break;
    case 'R':  // Reset
        reset_state();
        uart_send("Reset\n");
        break;
    case '?':  // Help
        uart_send("Commands: S, P, R, ?\n");
        break;
    default:
        uart_send("Unknown command\n");
}

Conditional (Ternary) Operator

Compact if-else in one line. Good for simple conditions.

uint8_t speed = (speed_sensor > 100) ? 255 : 128;
// If speed_sensor > 100, speed = 255, else speed = 128

// Equivalent if-else
uint8_t speed;
if (speed_sensor > 100) {
    speed = 255;
} else {
    speed = 128;
}

Good use: Simple substitutions

uint8_t max_val = (a > b) ? a : b;  // Clear and short

Bad use: Nested or complex

// AVOID
uint8_t result = (temp > 30) ? (humidity > 60 ? 3 : 2) : 1;

// Use if-else instead
uint8_t result;
if (temp > 30) {
    result = (humidity > 60) ? 3 : 2;
} else {
    result = 1;
}

Boolean Expressions

Conditions evaluate to true (1) or false (0).

uint8_t count = 5;

// These are equivalent
if (count > 0) { }      // Explicit
if (count) { }          // Implicit (non-zero = true)

// For clarity, be explicit
if (count != 0) { }     // Very clear

// For flags/booleans
uint8_t ready = 1;
if (ready) { }          // OK - it's a flag
if (ready == 1) { }     // Also OK

// Negation
if (!ready) { }         // NOT ready
if (ready == 0) { }     // Same thing

Early Return / Guard Clauses

Exit early to reduce nesting and improve readability.

// NESTED - Hard to follow
void process_command(uint8_t cmd) {
    if (cmd != 0) {
        if (system_ready) {
            if (!error_flag) {
                execute_command(cmd);
            }
        }
    }
}

// GUARD CLAUSES - Much clearer
void process_command(uint8_t cmd) {
    if (cmd == 0) return;              // Early exit
    if (!system_ready) return;         // Another early exit
    if (error_flag) return;            // Another
    
    execute_command(cmd);              // Main logic at top level
}

Real Embedded Example

uint8_t read_sensor_safe() {
    // Guard clauses: return early if conditions aren't met
    if (!power_enabled()) {
        return 0;
    }
    if (!sensor_initialized()) {
        return 0;
    }
    if (temperature_critical()) {
        return 0;
    }
    
    // If we get here, everything is OK
    return read_sensor();
}

Scope and Variable Visibility

Variables are only visible within their scope (curly braces).

uint8_t global_var = 1;  // Visible everywhere

void my_function() {
    uint8_t function_var = 2;  // Only visible in this function
    
    if (condition) {
        uint8_t block_var = 3;  // Only visible in this block
        // Can access: global_var, function_var, block_var
    }
    
    // Can't access block_var here - it's out of scope
    // CAN access global_var and function_var
}

// Can't access function_var or block_var here

Scope Advantages

// You can reuse variable names in different scopes
void sensor_reading() {
    uint8_t value = read_temp();
    process_temperature(value);
}

void voltage_reading() {
    uint8_t value = read_voltage();
    process_voltage(value);
}

// Each function has its own 'value' variable - no conflict

Common Flow Mistakes

  • Missing braces: Leads to unexpected behavior when multiple statements intended
  • Assignment vs comparison: if (x = 5) assigns, if (x == 5) compares
  • Forgetting break in switch: Code falls through to next case
  • Deep nesting: Hard to understand. Refactor with guard clauses
  • Relying on implicit true: if (counter) works but if (counter != 0) is clearer
// WRONG - Assignment in condition!
if (status = READY) {  // Now status is set to READY!
    // This runs even if status was something else
}

// RIGHT
if (status == READY) {
    // Check if status equals READY
}

// WRONG - Missing braces, confusing indentation
if (armed)
    disable_power();
    shutdown_safely();  // This ALWAYS runs!

// RIGHT
if (armed) {
    disable_power();
    shutdown_safely();
}

Performance Notes

In embedded systems, every instruction counts.

// Simple if-else is fast
if (flag) {
    action_a();
} else {
    action_b();
}

// Switch with many cases: compiler optimizes to jump table
// For many cases (10+), often faster than if-else chains
switch (state) {
    case 0: ... break;
    case 1: ... break;
    case 2: ... break;
    // ... many more cases
}

// Avoid deep nesting - makes branch prediction harder
// Instead, use guard clauses or switch statements

Summary

  • Use if for single conditions, if-else for alternatives
  • Use switch for checking one variable against many values
  • Always use braces to avoid mistakes
  • Use ternary operator for simple, clear expressions
  • Use guard clauses to reduce nesting
  • Be aware of scope—variables are only visible in their block
  • Use boolean logic to combine multiple conditions clearly

Mini-Projects to Strengthen Your Skills

Project 1: Traffic Light State Machine

Objective: Create a simple state machine that determines which traffic light to show based on a current state variable.

What you'll learn: How if-else chains represent simple decision logic in embedded systems.

Step by step:

  1. Create a uint8_t variable light_state (0 = red, 1 = yellow, 2 = green)
  2. Use if-else to check the state and set appropriate LEDs to HIGH or LOW
  3. Try all three states and verify the right LED lights up
// Define pins for each light
#define RED_PIN 2
#define YELLOW_PIN 3
#define GREEN_PIN 4

// Current light state
uint8_t light_state = 0;  // 0=red, 1=yellow, 2=green

// Showing the current light
uint8_t red_on = 0;
uint8_t yellow_on = 0;
uint8_t green_on = 0;

// Determine which light to show
if (light_state == 0) {
    red_on = 1;
    yellow_on = 0;
    green_on = 0;
} else if (light_state == 1) {
    red_on = 0;
    yellow_on = 1;
    green_on = 0;
} else if (light_state == 2) {
    red_on = 0;
    yellow_on = 0;
    green_on = 1;
} else {
    // Invalid state - turn all off
    red_on = 0;
    yellow_on = 0;
    green_on = 0;
}

// (In real code, you'd then write these to the actual pins)

Why this works: If-else chains let you map a state value to different outputs. This pattern appears everywhere in embedded systems: different modes, different behaviors based on a value, etc.

Project 2: Device Mode Selector with Switch

Objective: Use a switch statement to control different device behaviors based on a mode variable.

What you'll learn: Switch statements are cleaner than if-else when checking one variable against many values.

Step by step:

  1. Create a uint8_t device_mode (0=standby, 1=normal, 2=fast, 3=eco)
  2. Use a switch statement to set power consumption based on mode
  3. Create a uint8_t power_level and set it differently for each mode
uint8_t device_mode = 1;      // Current operating mode
uint8_t power_level = 0;      // How much power to use (0-255)
uint8_t led_brightness = 0;   // How bright to make status LED

switch (device_mode) {
    case 0:
        // Standby mode: minimal power
        power_level = 10;
        led_brightness = 10;
        break;
    
    case 1:
        // Normal mode: moderate power
        power_level = 128;
        led_brightness = 128;
        break;
    
    case 2:
        // Fast mode: maximum power
        power_level = 255;
        led_brightness = 255;
        break;
    
    case 3:
        // Eco mode: low power, optimized
        power_level = 50;
        led_brightness = 30;
        break;
    
    default:
        // Unknown mode: safe default
        power_level = 0;
        led_brightness = 0;
        break;
}

// (Now apply these settings to actual hardware)

Why this works: Each case represents a different mode. Without the break, it would "fall through" to the next case (usually a mistake). The default case catches invalid modes safely.

Project 3: Temperature-Based Fan Control

Objective: Use nested if-else and ternary operators to control fan speed based on temperature.

What you'll learn: Complex conditions using nesting and ternary operators for concise decision logic.

Step by step:

  1. Create a uint8_t temperature variable
  2. Use if-else to determine fan_status (0=off, 1=low, 2=medium, 3=max)
  3. Use a ternary operator to set warning_led (1 if temp > 40, else 0)
const uint8_t TEMP_OFF = 15;
const uint8_t TEMP_LOW = 20;
const uint8_t TEMP_MEDIUM = 28;
const uint8_t TEMP_MAX = 35;
const uint8_t TEMP_WARNING = 40;

uint8_t temperature = 22;

// Determine fan speed
uint8_t fan_status = 0;

if (temperature < TEMP_OFF) {
    fan_status = 0;  // Off
} else if (temperature < TEMP_LOW) {
    fan_status = 1;  // Low
} else if (temperature < TEMP_MEDIUM) {
    fan_status = 2;  // Medium
} else if (temperature < TEMP_MAX) {
    fan_status = 3;  // Max
} else {
    fan_status = 3;  // Stay at max if above max threshold
}

// Simple warning: true/false with ternary
uint8_t warning_led = (temperature > TEMP_WARNING) ? 1 : 0;

// (In real code, you'd apply these to actual hardware)

Why this works: If-else chains determine outputs, ternary operators handle simple true/false decisions. Together they give you flexible control over device behavior.

Key Takeaways

  • if statements: Execute code only if condition is true
  • if-else: Choose between two actions
  • else-if chains: Test multiple conditions in order
  • switch: Clean when checking one variable against many values
  • Use guard clauses to reduce nesting and improve readability
  • Always use braces to avoid subtle bugs
