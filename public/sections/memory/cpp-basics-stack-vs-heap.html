Stack vs Heap: Deep Dive

Difficulty Level: Advanced (Critical for Embedded)

This is THE most important lesson for embedded systems safety. In desktop programming, you can afford to spill memory or be sloppy. In embedded systems, a single memory mistake can crash your device, corrupt sensor data, or worst—cause safety-critical failures. This deep dive will make you understand memory at the level of the hardware.

The Memory Hierarchy

Before diving into stack vs heap, understand the memory landscape:

┌─────────────────────────────────┐
│      HIGH ADDRESSES             │
│  ┌─────────────────────────┐   │
│  │      STACK grows DOWN   │ ↓ │  
│  └─────────────────────────┘   │
│                                 │
│  ┌─────────────────────────┐   │
│  │      HEAP grows UP      │ ↑ │
│  └─────────────────────────┘   │
│                                 │
│  (uninitialized data segment)   │
│  (global/static data segment)   │
│          (TEXT - CODE)          │
└─────────────────────────────────┘
        LOW ADDRESSES

On a typical microcontroller (e.g., STM32F407 with 192KB RAM):

  • TEXT: Program code (stored in FLASH, not RAM)
  • Global/Static: Variables that exist for lifetime of program (e.g., `static uint8_t buffer[256]`)
  • HEAP: Dynamically allocated memory (grows upward)
  • STACK: Function calls and local variables (grows downward)
  • Unused: Free space between heap and stack (crash if they meet!)

The Stack: Fast, Automatic, Limited

What Lives on the Stack?

  • Local variables (declared inside functions)
  • Function parameters
  • Return addresses (where to jump back after function returns)
  • Saved registers (CPU state preservation)

How the Stack Works

uint8_t global_data[10];  // Global - exists forever

void function_a(int x) {
    uint8_t local_a = 42;
    function_b(x + 1);
}

void function_b(int y) {
    uint32_t local_b = 100;
    uint16_t local_c = 200;
    return;
}

// Memory diagram during execution:
// 
// STACK TOP (after function_b called):
// [Return address to function_a]
// [local_b = 100]
// [local_c = 200]
// [saved registers...]
// [Return address for function_a]
// [x parameter]
// [local_a = 42]
// STACK BASE

Stack Characteristics

Property Value
Allocation Automatic (just declare variable)
Deallocation Automatic (when function returns)
Speed FASTEST - single instruction (SP += size)
Size LIMITED (typically 1KB-4KB on microcontrollers)
Fragmentation NO fragmentation ever
Responsibility Compiler handles everything

Stack Example: Real Values

// STM32F407: 192KB total RAM
// Typical stack size: 8KB
// Typical heap size remaining: 184KB

// Safe function (small locals)
void read_sensor() {
    uint8_t sensor_id = 1;
    uint16_t reading = 0;
}

// Dangerous function (large local array)
void parse_data() {
    uint8_t buffer[4096];
}
}

// Correct approach for large buffers
uint8_t global_buffer[4096];
void parse_data() {
    global_buffer[0] = read_byte();
}
⚠️ RULE #1: Never declare large arrays on the stack. Use global/static arrays instead. Large local arrays are the #1 cause of embedded system crashes.

The Heap: Flexible, Slow, Large

What Lives on the Heap?

  • Dynamically allocated memory from `new`
  • Anything requested at runtime (not known at compile time)
  • Data structures** that grow/shrink during execution

Heap Allocation Example

// Allocate on heap (slow, but flexible)
uint32_t *data = new uint32_t[100];
data[50] = 12345;
delete[] data;

// Problems:
// 1. If you forget delete, memory leaks
// 2. If you delete twice, crash
// 3. If you use after delete, corruption

Heap Characteristics

Property Value
Allocation Manual (must call `new`)
Deallocation Manual (must call `delete`) - ERROR-PRONE!
Speed SLOW - malloc/free involves complex bookkeeping
Size LARGE (but shared with stack!)
Fragmentation YES - repeated alloc/dealloc fragments memory
Responsibility Programmer's responsibility (danger!)

Stack Overflow: The Silent Killer

When Stacks Collide

Stack grows downward, heap grows upward. If they meet, memory corruption:

// Scenario: STM32F407 with 192KB RAM
// Stack starts at top (0x20030000)
// Heap starts at bottom (0x20000000)
// Total space: 192KB

void recursive_function(int depth) {
    uint8_t buffer[1024];
    if (depth < 200) {
        recursive_function(depth + 1);
    }
    }
}

// Result: CRASH
// Stack pointer eventually < heap pointer
// Overwrite heap data or fall into wild memory

Detection: Check Free Memory

// Linker script defines these
extern uint32_t _heap_start;
extern uint32_t _heap_end;

uint32_t get_free_memory() {
    extern uint32_t __get_MSP(void);
    uint32_t sp = __get_MSP();
    uint32_t heap_ptr = (uint32_t)(&_heap_end);
    
    if (sp > heap_ptr) {
        return sp - heap_ptr;
    }
    return 0;
    
    if (sp > heap_ptr) {
        return sp - heap_ptr;  // Free space
    }
    return 0;  // Collision!
}
⚠️ CRITICAL: Stack overflow is silent and deadly. The microcontroller doesn't crash immediately—it silently corrupts memory. Weeks later, a sensor reads wrong. A motor runs wrong. The device is unreliable.

Avoiding Heap in Embedded Systems

Why Most Embedded Code AVOIDS `new` and `delete`

In mission-critical embedded systems, dynamic allocation is often forbidden:

// ❌ STYLE 1: Embedded system without proper memory management
void process_data(uint8_t *raw_data) {
}

// ✅ STYLE 2: Proper embedded approach (no allocation)
uint8_t processed[MAX_BUFFER];
void process_data(uint8_t *raw_data) {
}

When Allocation is Unavoidable

If you MUST allocate dynamically, use a pool allocator instead of raw malloc:

// Pool allocator: pre-allocate blocks, reuse them
class MemoryPool {
    static const int BLOCK_SIZE = 256;
    static const int NUM_BLOCKS = 10;
    uint8_t blocks[NUM_BLOCKS][BLOCK_SIZE];
    bool block_used[NUM_BLOCKS];
    
public:
    void* allocate() {
        for (int i = 0; i < NUM_BLOCKS; i++) {
            if (!block_used[i]) {
                block_used[i] = true;
                return blocks[i];
            }
        }
        return nullptr;  // No free blocks
    }
    
    void free(void* ptr) {
        for (int i = 0; i < NUM_BLOCKS; i++) {
            if ((void*)blocks[i] == ptr) {
                block_used[i] = false;
                return;
            }
        }
    }
};

// Usage
MemoryPool pool;
uint8_t *buffer = (uint8_t*)pool.allocate();
if (buffer) {
    // Use buffer
    pool.free(buffer);
}

Real Embedded Memory Models

Model 1: No Dynamic Allocation (Safest)

// Everything is static/global or stack-based
#define NUM_SENSORS 4
#define BUFFER_SIZE 256

struct SensorData {
    uint16_t temperature;
    uint16_t humidity;
} sensors[NUM_SENSORS];  // Fixed array of sensors

uint8_t uart_buffer[BUFFER_SIZE];  // Fixed UART buffer

void main_loop() {
    for (int i = 0; i < NUM_SENSORS; i++) {
        sensors[i].temperature = read_temp_sensor(i);
    }
    transmit_uart(uart_buffer, BUFFER_SIZE);
}

// Pro: No fragmentation, no memory leaks, predictable memory use
// Con: Can't grow/shrink at runtime, fixed at compile-time

Model 2: Limited Allocation with Pools

// Pre-allocate pools, reuse blocks
class PacketMemory {
    struct Packet { uint8_t data[64]; } packets[20];
    bool free_list[20];
    
public:
    Packet* allocate_packet() {
        for (int i = 0; i < 20; i++) {
            if (free_list[i]) {
                free_list[i] = false;
                return &packets[i];
            }
        }
        return nullptr;
    }
    
    void free_packet(Packet *p) {
        int index = (p - packets);
        free_list[index] = true;
    }
};

PacketMemory packet_pool;

void handle_message() {
    Packet *p = packet_pool.allocate_packet();
    if (!p) return;  // No memory available
    
    receive_data(p->data);
    process_packet(p);
    packet_pool.free_packet(p);
}

Model 3: Static Allocation with Ring Buffers

For streaming data (UART, SPI, DMA):

// Ring/Circular buffer: reuses same memory
class RingBuffer {
    static const int SIZE = 256;
    uint8_t buffer[SIZE];
    int write_idx = 0;
    int read_idx = 0;
    
public:
    void write(uint8_t byte) {
        buffer[write_idx] = byte;
        write_idx = (write_idx + 1) % SIZE;  // Wrap around
    }
    
    uint8_t read() {
        uint8_t byte = buffer[read_idx];
        read_idx = (read_idx + 1) % SIZE;
        return byte;
    }
    
    bool is_empty() { return write_idx == read_idx; }
    bool is_full() { return (write_idx + 1) % SIZE == read_idx; }
};

RingBuffer uart_rx;  // Receives data endlessly without growing

Memory Debugging Techniques

Technique 1: Stack Canary

// Place a known value on stack, check if overwritten
void check_stack_overflow() {
    uint32_t canary = 0xDEADBEEF;
    }
}

Technique 2: Memory Sentinel Bytes

// Allocate more memory than needed, fill boundaries with sentinel
uint8_t *safe_allocate(int size) {
    uint8_t *ptr = (uint8_t*)malloc(size + 2);
    ptr[0] = 0xAA;
    ptr[size + 1] = 0xBB;
    return &ptr[1];
}

void check_buffer_overflow(uint8_t *ptr) {
    if (ptr[-1] != 0xAA || ptr[expected_size] != 0xBB) {
        error_handler();
    }
}

Best Practices for Embedded Memory

  • Prefer stack: Variables are deallocated automatically
  • Use static buffers: For fixed-size data (sensors, protocols)
  • Avoid large stack allocations: Keep functions' local data < 1KB
  • Avoid dynamic allocation: Use pools instead of malloc/new
  • Check free memory: Monitor available RAM (especially with networking)
  • Use const where possible: Store lookup tables in FLASH, not RAM
  • Profile memory: Compile with `-fstack-usage` to find stack hogs
  • Test on real hardware: Simulators don't catch many memory bugs

The Golden Rules

✅ Memory Safety Rules:
  1. Know your limits: How much RAM? How much stack? MEASURE it.
  2. Avoid large stack allocations: Max 256 bytes per function unless you're sure.
  3. No dynamic allocation: Use pools if you must allocate.
  4. Debug memory: Add canaries, sentinels, monitors.
  5. Crash safely: When memory runs out, enter safe mode, not undefined behavior.

Summary

Stack and Heap serve different purposes:

  • Stack: Fast, automatic, small, safe (if used correctly)
  • Heap: Large, manual, slow, dangerous (if misused)

In embedded systems, minimize heap usage and design around static allocation. The systems that never crash are the ones that never surprise their memory manager.
