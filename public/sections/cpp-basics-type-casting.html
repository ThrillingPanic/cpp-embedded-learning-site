Type Casting & Conversions

Difficulty Level: Fundamental

Type casting is how you convert one data type to another. In embedded systems, this is critical because you're constantly working with hardware that expects specific types: ADC readings come in as uint16_t, but you might need to work with them as floats for filtering. A sensor returns bytes, but you need to interpret them as signed values.

Why Type Casting Matters

Embedded Reality: Hardware is type-strict. A GPIO register expects uint32_t. A temperature sensor returns 12-bit values. A UART buffer is uint8_t. You WILL encounter type mismatches, and understanding casting prevents subtle bugs.

⚠️ Warning: Incorrect casting can silently corrupt data. A -10 stored in uint8_t becomes 246. A float truncated to int loses its decimal part. These bugs are insidious because the code compiles fine.

1. Implicit Conversions (Type Coercion)

C++ automatically converts types in certain situations. This is convenient but dangerous in embedded code because you might not notice data loss.

Widening Conversions (Safe)

Expanding from smaller to larger type—no data loss:

uint8_t sensor_reading = 42;
uint16_t expanded = sensor_reading;  // OK: 8-bit → 16-bit

When you expand from uint8_t to uint16_t, the value 42 is preserved perfectly. The larger container has plenty of room. Nothing is lost:

uint32_t more_space = expanded;  // OK: 16-bit → 32-bit

Same principle applies to signed integers:

int8_t temp = -5;
int16_t safe = temp;  // Fine: still -5
int32_t safer = safe; // Still -5

Widening is always safe. You're moving from a small box to a bigger box—nothing gets lost in the move.

Narrowing Conversions (Dangerous!)

Shrinking from larger to smaller type—DATA LOSS:

uint16_t adc_value = 300;
uint8_t truncated = adc_value;  // Becomes 44 (300 & 0xFF)!
// Upper bits thrown away. Value corrupted.

int16_t sensor = 260;
int8_t broken = sensor;  // Becomes 4 (truncation)
// In embedded, this causes real-world failures

Why this happens: uint16_t value 300 = 0x012C. Only the lower byte (0x2C = 44) fits in uint8_t. The rest is discarded.

Signed/Unsigned Mixing (Very Dangerous!)

Mixing signed and unsigned is a classic bug:

int8_t temp = -10;
uint8_t positive = temp;

int16_t sensor_reading = -50;
if (sensor_reading < 200) {
Embedded Rule: Never implicitly convert between signed and unsigned. Always cast explicitly so future readers understand your intent.

2. Explicit C-Style Casts

Traditional C-style casting with parentheses. Simple but dangerous because it forces any conversion:

uint16_t adc = 300;
uint8_t small = (uint8_t)adc;

float voltage = 3.3;
uint16_t digital = (uint16_t)voltage;

const uint32_t *hardware_reg = (const uint32_t *)0x40000000;
uint32_t *mutable_ptr = (uint32_t *)hardware_reg;

Problems with C-style casts:

  • Hard to search for (just looks like parentheses)
  • Too powerful—can cast almost anything to anything
  • Hides bugs during code review

3. C++ Casts (Better!)

C++ provides four specific casts. Use these instead of C-style casts in modern embedded code:

static_cast - Safe Type Conversions

For conversions that the compiler knows are safe (or intentionally lossy):

// Converting between numeric types
uint16_t adc_value = 1024;
float voltage = static_cast(adc_value) * 3.3 / 4096.0;

// Converting between class types (if valid conversion exists)
int sensor_value = 42;
char ascii_digit = static_cast('0' + sensor_value);

// Pointers to base/derived class
class Sensor { public: virtual void read() {} };
class TemperatureSensor : public Sensor {};
Sensor *base_ptr = new TemperatureSensor();
TemperatureSensor *derived = static_cast(base_ptr);

reinterpret_cast - Bit-Level Reinterpretation

For pointer/reference casting where you're saying "interpret these bits as a different type". Dangerous but necessary for hardware access:

// Hardware register access (this is where reinterpret_cast shines!)
uint32_t *gpio_port_a = reinterpret_cast(0x40020000);
uint32_t port_state = *gpio_port_a;

// Interpreting buffer data as a struct
struct SensorData { uint16_t temp; uint16_t humidity; };
uint8_t raw_buffer[4] = { 0x10, 0x01, 0x50, 0x00 };
SensorData *data = reinterpret_cast(raw_buffer);
// Now data->temp accesses first 2 bytes, data->humidity accesses last 2

// Converting int to pointer (used in bootloader memory copying)
void *buffer = reinterpret_cast(0x08000000);  // Flash memory address
⚠️ Critical: reinterpret_cast is how you access hardware registers. It's essential for embedded, but dangerous if misused. Always comment why you're doing it.

const_cast - Add/Remove const

Remove const from a pointer or reference (usually a bad idea, but sometimes necessary):

void old_c_function(uint32_t *data);
const uint32_t hw_register = 0x12345678;
old_c_function(const_cast(&hw_register));

volatile uint32_t *sensor_port = reinterpret_cast(0x40021000);
const uint32_t snapshot = *sensor_port;

dynamic_cast - Runtime Type Checking

Safely cast pointers/references in inheritance hierarchies. Returns nullptr on failure:

// Probably NOT used in resource-constrained embedded code (requires RTTI)
class Sensor { public: virtual ~Sensor() {} };
class TempSensor : public Sensor {};
class HumiditySensor : public Sensor {};

Sensor *sensor = new TempSensor();
TempSensor *temp = dynamic_cast(sensor);  // Safe!
if (temp) { /* Yes, it's a TempSensor */ }
else { /* No, it's something else */ }

// Returns nullptr if cast fails
HumiditySensor *humidity = dynamic_cast(sensor);
if (!humidity) { /* Not a humidity sensor */ }
Note: dynamic_cast requires RTTI (Run-Time Type Information), which adds overhead. Most embedded systems disable RTTI to save code space. Use static_cast if you know the type at compile-time.

4. Real Embedded Examples

Example 1: ADC to Voltage Conversion

// Raw 12-bit ADC value (0-4095)
uint16_t adc_raw = read_adc_channel(0);

// Convert to voltage (0-3.3V)
float reference_voltage = 3.3f;
float adc_max = 4095.0f;
float voltage = static_cast(adc_raw) * reference_voltage / adc_max;

// Problem: float is slow on microcontrollers without FPU
// Better: keep as integer math
uint16_t millivolts = (adc_raw * 3300) / 4095;  // Result in mV

Example 2: Parsing Multi-Byte Sensor Data

// Sensor sends 4 bytes: [status_byte][temp_high][temp_low][crc]
uint8_t sensor_buffer[4] = { 0x01, 0x19, 0x5C, 0x42 };

// Method 1: Manually extract
uint8_t status = sensor_buffer[0];
int16_t temp = static_cast((sensor_buffer[1] << 8) | sensor_buffer[2]);
// temp = 0x195C = 6492 (raw sensor units)

// Method 2: Reinterpret as struct (careful with alignment!)
struct SensorReading {
    uint8_t status;
    int16_t temperature;  // Note: byte order matters!
    uint8_t crc;
};
// Dangerous! Assumes buffer layout matches struct
// SensorReading *data = reinterpret_cast(sensor_buffer);

Example 3: Hardware Register Access

// GPIO Port A Control Register at memory address 0x40020000
volatile uint32_t *gpio_port_a = reinterpret_cast(0x40020000);

// Set GPIO pin 5 as output (write 0x20 to lower byte)
*gpio_port_a = 0x20;

// Alternatively, use volatile uint32_t& for references
volatile uint32_t &gpio_control = *reinterpret_cast(0x40020000);
gpio_control |= 0x20;  // Set bit 5

Example 4: Type-Safe Status Code Enum

// Define status codes
enum class SensorStatus : uint8_t {
    OK = 0x00,
    ERROR = 0x01,
    TIMEOUT = 0x02,
    OVERFLOW = 0x03
};

// Hardware returns raw uint8_t
uint8_t status_byte = read_hardware_register();

// Safe conversion
SensorStatus status = static_cast(status_byte);

// Safer approach: validate the conversion
if (status_byte <= 0x03) {
    SensorStatus status = static_cast(status_byte);
} else {
}

5. Casting Pitfalls & Solutions

Pitfall 1: Truncation Without Notice

// Silent data loss
uint32_t large_value = 1000000;
uint16_t small = large_value;  // Compiles! But value becomes 64896 (overflow)

// Better: explicit cast shows intent
uint16_t small = static_cast(large_value & 0xFFFF);

Pitfall 2: Signed/Unsigned Confusion

// Sensor reads -5°C, stored as signed int8_t
int8_t temperature = -5;

// Need to send over UART as unsigned byte
uint8_t uart_data = static_cast(temperature);  // Becomes 251

// Receiver must know to interpret as signed
int8_t received_temp = static_cast(uart_data);  // Back to -5

Pitfall 3: Float to Int Truncation

// Incorrect: data loss
float sensor_value = 3.87;
int scaled = sensor_value * 100;  // Becomes 387 (usually)

// Better: be explicit
int scaled = static_cast(sensor_value * 100);

// Best for embedded: scale to integer first, avoid float
uint16_t raw_adc = 987;
uint16_t millivolts = (raw_adc * 3300) / 4095;  // Integer-only, exact

Best Practices for Embedded Casting

  • Prefer static_cast: It's the default choice for numeric conversions and inheritance.
  • Use reinterpret_cast for hardware: Accessing memory-mapped registers requires it. Always comment why.
  • Avoid implicit conversions: Use explicit casts so readers understand your intent.
  • Avoid C-style casts: They're too powerful and hide mistakes.
  • Validate conversions: If casting from uint32_t to uint8_t, check first that value fits.
  • Document byte order: When casting between structs and raw buffers, specify endianness.
  • Use const_cast rarely: If you're casting away const, question why. Usually indicates design issue.
  • Test on actual hardware: Casting behavior can differ on different architectures.

Summary

Type casting is unavoidable in embedded programming. The key is understanding:

  • Implicit conversions are convenient but dangerous (narrowing = data loss)
  • static_cast handles numeric conversions and safe class conversions
  • reinterpret_cast accesses hardware registers and reinterprets memory
  • const_cast removes const (rare, usually unnecessary)
  • dynamic_cast is for runtime type checking with inheritance (usually too expensive for embedded)

Always be explicit about casting and documented about your intent. Future you (and your teammates) will thank you.

✅ Key Takeaway: In embedded systems, every cast is a potential data loss point. Make them explicit, validate them, and comment why you're doing it. The compiler will catch type errors if you let it—don't force conversions with casts unless absolutely necessary.
