Why Debugging is Different in Embedded Systems

On your laptop, if something crashes, you get an error message. An exception is thrown. The program stops. You see a stack trace. This makes debugging relatively straightforward—the system tells you what went wrong.

On a microcontroller, if something goes wrong, the device just... keeps running. Or it crashes silently. Or it behaves erratically. There's no error message. No stack trace. No exception handler. The device is often out in the field, inaccessible, running in a loop forever.

This is the fundamental challenge: In embedded systems, **you must design debugging into your code from the start.** You can't rely on the system to tell you what went wrong. You must build the ability to inspect what's happening while the program runs.

Embedded Debugging Principle: If you can't see what your code is doing, you can't fix it when it breaks. Always design systems so you can observe them in action.

Real example: A temperature sensor on an Arduino reads 255°C when it should read 25°C. The microcontroller has no way to report this—it just keeps running with the wrong value. If you haven't built in debugging, you'll spend hours wondering why your cooling system never activates. With debugging, you send the sensor value to your computer via serial and immediately see the problem.

Another example: Your motor controller crashes after 3 hours of operation. The device just stops responding. No error message. Without debugging instrumentation, you'll never know if it's:

  • Stack overflow (function calls too deep)
  • Memory leak (heap exhausted)
  • Infinite loop (watchdog timeout)
  • Interrupt handler crash
  • Hardware failure

With proper debugging, you'd have logged enough information to pinpoint the cause in seconds.

Serial Debugging:printf Debugging

The simplest and most powerful debugging technique: Send data from your microcontroller to your computer via serial (UART). Then you can observe exactly what's happening.

On Arduino, this is `Serial.println()`. On desktop C++, this is `printf()`. The principle is identical: output information at runtime to diagnose problems.

Setting Up Serial Output (Arduino)

void setup() {
    Serial.begin(9600);  // Start serial communication at 9600 baud
    Serial.println("System initialized");
}

void loop() {
    uint16_t sensor_reading = analogRead(A0);
    Serial.print("Sensor: ");
    Serial.println(sensor_reading);
    delay(1000);
}

What happens: Your Arduino sends text across the USB cable to your computer. You open the Serial Monitor (Tools → Serial Monitor in Arduino IDE) and watch the data stream in real-time. This is printf-style debugging.

Strategic Debugging Points

Don't just print randomly. Place debugging statements strategically at critical points:

void process_sensor_data(uint16_t raw_value) {
    Serial.print("INPUT: ");
    Serial.println(raw_value);
    
    uint16_t converted = raw_value * 5 / 1024;  // Convert ADC to voltage
    Serial.print("CONVERTED: ");
    Serial.println(converted);
    
    if (converted > DANGER_THRESHOLD) {
        Serial.println("WARNING: DANGER THRESHOLD EXCEEDED");
        trigger_alarm();
    }
    
    Serial.print("OUTPUT: ");
    Serial.println(converted);
}

This lets you trace execution: You see the input value, the conversion, the decision point, and the output. If something goes wrong, you immediately see where the logic broke.

Debugging Levels: Control Output Volume

In production code, you don't want debug output cluttering everything. Use debug levels:

#define DEBUG_LEVEL 2  // 0=off, 1=errors, 2=warnings, 3=verbose

void debug_print(int level, const char *msg) {
    if (level <= DEBUG_LEVEL) {
        Serial.println(msg);
    }
}

void loop() {
    debug_print(1, "Starting main loop");
    
    uint16_t sensor = analogRead(A0);
    debug_print(3, "Sensor read complete");  // Verbose only
    
    if (sensor > THRESHOLD) {
        debug_print(1, "ERROR: Sensor out of range");
    }
}

Benefits: You can control how much debugging output you see with a single #define. In production, set DEBUG_LEVEL to 0 for no output. During development, set it to 3 for everything.

⚠️ Warning: Serial Output Performance

Serial communication is slow! Sending text at 9600 baud takes time. Each character takes ~1ms. Heavy debugging output can cause timing problems in real-time systems. Keep serial debugging brief or use higher baud rates (115200, 230400).

LED Debugging: Visual Diagnostics Without Serial

Sometimes you can't access serial output. Your device is deployed on a robot miles away. Or you need real-time feedback but serial is too slow. That's when you use LEDs.

LED blinking is the embedded systems equivalent of `assert(true)`. It's how you know the code reached a certain point.

Basic LED Debugging

#define STATUS_LED 13

void check_sensor() {
    int value = analogRead(A0);
    
    digitalWrite(STATUS_LED, HIGH);  // Code reached this point
    delay(100);
    digitalWrite(STATUS_LED, LOW);
    
    if (value < THRESHOLD) {
        digitalWrite(STATUS_LED, HIGH);
        delay(500);
        digitalWrite(STATUS_LED, LOW);
    }
}

What you observe:

  • LED blinks 100ms? Code executed the check
  • LED stays on for 500ms? Threshold condition was true
  • LED doesn't blink at all? Function never ran (code is stuck elsewhere)

Blink Codes: Multi-Blink Patterns

Different error conditions get different blink patterns:

#define STATUS_LED 13

void blink_pattern(int count) {
    for (int i = 0; i < count; i++) {
        digitalWrite(STATUS_LED, HIGH);
        delay(100);
        digitalWrite(STATUS_LED, LOW);
        delay(100);
    }
    delay(500);  // Gap between patterns
}

void setup() {
    pinMode(STATUS_LED, OUTPUT);
    
    if (!init_sensor()) {
        blink_pattern(1);  // 1 blink = sensor error
        while (1);
    }
    
    if (!init_communication()) {
        blink_pattern(2);  // 2 blinks = communication error
        while (1);
    }
    
    blink_pattern(3);  // 3 blinks = all systems OK
}

On startup: You see blinking patterns that tell you which system failed. No need for serial output—just watch the LED.

Real-World LED Debugging Pattern

// Heartbeat LED - blinks every loop iteration
void main_loop() {
    toggle_heartbeat_led();  // Blink once per iteration
    
    process_sensor();
    update_control();
    handle_communication();
}

Purpose: You can see the device is still running (LED blinks). If the LED stops, you know the system is frozen. This is your first diagnostic tool when something fails.

Production Reality: Industrial embedded systems always have status LEDs—not just for debugging, but for operators to understand system state. Red = error, green = healthy, yellow = warning. The same tool you use for debugging becomes a user-facing indicator.

Hardware Debugging: Breakpoints and Stepping

When printf debugging isn't enough, you need a hardware debugger. This lets you pause execution, inspect variables, and step through code line-by-line.

What is a Hardware Debugger?

Most modern microcontrollers have a built-in debugging interface (JTAG, SWD). Connect your development computer to this interface with a special cable (like STLink on STM32), and you can:

  • Set breakpoints: Pause execution at specific lines
  • Step through code: Execute one line at a time, inspecting variables
  • Inspect memory: Read RAM and registers while paused
  • View call stack: See how you got to the current function
  • Trace execution: Record every instruction executed (on advanced debuggers)

Using GDB (GNU Debugger) with STM32

// Compile with debug symbols
arm-none-eabi-gcc -g -O0 firmware.c -o firmware.elf

// Start OpenOCD (communicates with STLink)
openocd -f interface/stlink.cfg -f target/stm32f4x.cfg

// Connect GDB in another terminal
arm-none-eabi-gdb firmware.elf
(gdb) target remote localhost:3333
(gdb) break main                    // Set breakpoint in main()
(gdb) continue                       // Run until breakpoint
(gdb) print variable_name           // Inspect variable
(gdb) next                           // Step to next line
(gdb) step                           // Step into function
(gdb) backtrace                      // Show call stack

When to Use Hardware Debugging

Use hardware debugging when:

  • Serial debugging is too slow (real-time code)
  • Memory corruption is suspected (can't use printf safely)
  • Interrupt timing is critical (can't add delays)
  • Analyzing race conditions (multiple concurrent processes)
  • Exploring unfamiliar code (need to step through)

Don't use hardware debugging when:

  • Simple printf debugging would suffice (faster to add print statements)
  • The bug is timing-dependent (debugger changes timing)
  • The device is deployed remotely (no debugger access)
⚠️ The Heisenbug Problem

Adding a breakpoint changes timing. Pausing execution affects interrupt handlers. A bug that appears during development might disappear when you stop debugging to add print statements. This is called a "Heisenbug" (named after Heisenberg's uncertainty principle). Combat this by using non-invasive debugging (logging, LEDs) as much as possible.

Common Embedded Bugs and How to Catch Them

Bug #1: Uninitialized Variables

Symptom: Behavior is random on startup, then stable

// BUG
uint16_t sensor_reading;  // Contains garbage!
if (sensor_reading > THRESHOLD) {
    trigger_alarm();  // Might trigger randomly on startup
}

// FIX
uint16_t sensor_reading = 0;  // Always initialize

How to catch: Add initialization checks in your debug output. Print every variable on startup to verify it's at expected value.

Bug #2: Stack Overflow

Symptom: Device crashes after random amount of time; behavior is unpredictable

// BUG - huge local array
void process_image() {
    uint8_t buffer[256 * 256];  // 65KB on stack!
}

// FIX - use static or global
uint8_t buffer[256 * 256];
void process_image() {
    // Use buffer
}

How to catch: Compile with `-fstack-usage` to see stack usage per function. Monitor free stack with `get_free_memory()`. Set canary values on stack and check if overwritten.

Bug #3: Integer Overflow

Symptom: Calculations suddenly produce wrong values; behavior looks like a sudden jump

// BUG
uint8_t counter = 255;
counter++;  // Counter becomes 0!

// FIX
uint16_t counter = 255;
counter++;  // Counter becomes 256 (safe)

How to catch: Add assertions for expected ranges. Print values before and after arithmetic operations.

Bug #4: Memory Leak

Symptom: Device works fine at first, then gradually slows down or crashes

// BUG - allocate but never free
void handle_packet() {
    uint8_t *buffer = malloc(1024);
    process(buffer);
    return;  // Leaked!
}

// FIX
void handle_packet() {
    uint8_t *buffer = malloc(1024);
    process(buffer);
    free(buffer);
}

How to catch: Track memory allocation. Log every malloc/free. Monitor total allocated bytes over time. Stop main loop if total allocation exceeds threshold.

Bug #5: Infinite Loop / Deadlock

Symptom: Device becomes completely unresponsive

// BUG
while (waiting_for_data) {
    // Stuck here forever if data never arrives
}

// FIX - add timeout
uint32_t start = millis();
while (waiting_for_data && millis() - start < TIMEOUT) {
    // Will eventually exit
}
if (waiting_for_data) {
    Serial.println("ERROR: Timeout waiting for data");
}

How to catch: Add watchdog timer. Implement timeout for every wait operation. Use heartbeat LED to detect freezing.

Bug #6: Off-by-One Array Access

Symptom: Mysterious behavior; random crashes or unexpected data corruption

// BUG
uint8_t buffer[10];
for (int i = 0; i <= 10; i++) {  // Should be i < 10
    buffer[i] = i;  // Writes past end of array!
}

// FIX
for (int i = 0; i < 10; i++) {
    buffer[i] = i;
}

How to catch: Use bounds-checking in debug builds. Initialize memory with distinctive patterns (0xAA, 0x55) and check if they change unexpectedly.

Key Takeaways

  • Embedded systems won't tell you when something's wrong—design debugging in from the start
  • Serial debugging (printf) is the fastest way to diagnose most problems
  • LED blinking provides visual diagnostics without serial connection
  • Hardware debugging (breakpoints, stepping) is powerful but can change timing
  • Always initialize variables, check bounds, and validate inputs
  • Add timeout to every wait operation—prevent infinite loops
  • Use watchdog timers and heartbeat LEDs in production systems
  • Most embedded bugs fall into predictable categories—learn to recognize patterns

Mini-Projects to Strengthen Your Skills

Project 1: Serial Debug Output System

Objective: Build a templated debugging system with debug levels and formatted output.

Requirements:

  • Create debug levels (ERROR=1, WARN=2, INFO=3, DEBUG=4)
  • Implement a debug_print() function that respects current debug level
  • Print sensor readings at different debug levels
  • Toggle debug level with a serial command
  • Measure timing impact of debug output
#define DEBUG_LEVEL 3  // Set to 0-4

#define ERROR_LEVEL 1
#define WARN_LEVEL 2
#define INFO_LEVEL 3
#define DEBUG_LEVEL_LOG 4

void debug_print(int level, const char *msg) {
    if (level <= DEBUG_LEVEL) {
        Serial.print("[");
        Serial.print(millis());
        Serial.print("] ");
        
        switch (level) {
            case ERROR_LEVEL: Serial.print("ERROR: "); break;
            case WARN_LEVEL: Serial.print("WARN: "); break;
            case INFO_LEVEL: Serial.print("INFO: "); break;
            case DEBUG_LEVEL_LOG: Serial.print("DEBUG: "); break;
        }
        
        Serial.println(msg);
    }
}

void setup() {
    Serial.begin(115200);
    debug_print(INFO_LEVEL, "System startup");
}

void loop() {
    uint16_t sensor = analogRead(A0);
    
    debug_print(DEBUG_LEVEL_LOG, "Reading sensor...");
    debug_print(INFO_LEVEL, "Sensor value: " + String(sensor));
    
    if (sensor > 900) {
        debug_print(WARN_LEVEL, "Sensor reading high!");
    }
    
    delay(1000);
}

Key Learning: Debug output should be formatted consistently with timestamps and level indicators. This makes it easy to filter and analyze logs.

Project 2: LED Blink Codes for Error Reporting

Objective: Implement a multi-LED diagnostic system with distinctive blink codes for different errors.

Requirements:

  • Define 5 different system states (OK, ERROR, WARNING, CRITICAL, TIMEOUT)
  • Assign unique blink patterns to each state
  • Create a state machine that changes LED pattern based on conditions
  • Simulate sensor failures and watch LED patterns change
  • Design patterns that are visually distinct (not confused with each other)
#define STATUS_LED 13
#define LED_ON_TIME 100
#define LED_OFF_TIME 100
#define GAP_TIME 500

enum SystemState {
    STATE_OK,
    STATE_WARNING,
    STATE_ERROR,
    STATE_CRITICAL,
    STATE_TIMEOUT
};

SystemState current_state = STATE_OK;
uint32_t last_blink = 0;

void blink_pattern(int count, int delay_ms) {
    for (int i = 0; i < count; i++) {
        digitalWrite(STATUS_LED, HIGH);
        delay(LED_ON_TIME);
        digitalWrite(STATUS_LED, LOW);
        delay(LED_OFF_TIME);
    }
    delay(delay_ms);
}

void update_led() {
    if (millis() - last_blink < 3000) return;  // Pattern every 3 seconds
    last_blink = millis();
    
    switch (current_state) {
        case STATE_OK:
            blink_pattern(1, GAP_TIME);  // 1 blink = OK
            break;
        case STATE_WARNING:
            blink_pattern(2, GAP_TIME);  // 2 blinks = warning
            break;
        case STATE_ERROR:
            blink_pattern(3, GAP_TIME);  // 3 blinks = error
            break;
        case STATE_CRITICAL:
            blink_pattern(5, GAP_TIME);  // 5 blinks = critical
            break;
        case STATE_TIMEOUT:
            digitalWrite(STATUS_LED, HIGH);  // Solid on = timeout
            break;
    }
}

void loop() {
    update_led();
    
    // Simulate conditions
    uint16_t sensor = analogRead(A0);
    
    if (sensor < 100) {
        current_state = STATE_CRITICAL;
    } else if (sensor < 200) {
        current_state = STATE_ERROR;
    } else if (sensor < 500) {
        current_state = STATE_WARNING;
    } else {
        current_state = STATE_OK;
    }
}

Key Learning: Pattern distinctiveness is critical. 1 vs 2 vs 3 blinks can be confused visually. Use different patterns (short/long, counting, solid) for maximum clarity.

Project 3: Memory Monitoring and Leak Detection

Objective: Track dynamic memory allocation and detect leaks.

Requirements:

  • Create wrapper functions track_malloc() and track_free()
  • Log all allocations with size and timestamp
  • Display current memory usage periodically
  • Detect leaks (allocations without corresponding free)
  • Alert if memory usage exceeds threshold
#define MAX_ALLOCATIONS 20
#define MEMORY_WARNING_THRESHOLD 1500

struct Allocation {
    void *ptr;
    uint16_t size;
    uint32_t timestamp;
    bool active;
};

Allocation tracked[MAX_ALLOCATIONS];
int allocation_count = 0;
uint32_t total_allocated = 0;

void* track_malloc(uint16_t size) {
    void *ptr = malloc(size);
    
    if (ptr && allocation_count < MAX_ALLOCATIONS) {
        tracked[allocation_count].ptr = ptr;
        tracked[allocation_count].size = size;
        tracked[allocation_count].timestamp = millis();
        tracked[allocation_count].active = true;
        allocation_count++;
        
        total_allocated += size;
        
        Serial.print("ALLOC: ");
        Serial.print(size);
        Serial.print(" bytes at ");
        Serial.println((unsigned long)ptr);
        
        if (total_allocated > MEMORY_WARNING_THRESHOLD) {
            Serial.println("WARNING: Memory threshold exceeded!");
        }
    }
    
    return ptr;
}

void track_free(void *ptr) {
    for (int i = 0; i < allocation_count; i++) {
        if (tracked[i].ptr == ptr && tracked[i].active) {
            tracked[i].active = false;
            total_allocated -= tracked[i].size;
            
            Serial.print("FREE: ");
            Serial.print(tracked[i].size);
            Serial.print(" bytes, total now: ");
            Serial.println(total_allocated);
            
            free(ptr);
            return;
        }
    }
    
    Serial.println("ERROR: Free called on unknown pointer");
}

void print_memory_stats() {
    Serial.println("\n=== Memory Statistics ===");
    Serial.print("Total allocated: ");
    Serial.print(total_allocated);
    Serial.println(" bytes");
    
    int active_allocs = 0;
    for (int i = 0; i < allocation_count; i++) {
        if (tracked[i].active) active_allocs++;
    }
    
    Serial.print("Active allocations: ");
    Serial.println(active_allocs);
    
    Serial.println("=========================\n");
}

Key Learning: Tracking memory at runtime reveals leaks immediately. This system can be extended with leak detection (allocations not freed after timeout) or histogram analysis.
