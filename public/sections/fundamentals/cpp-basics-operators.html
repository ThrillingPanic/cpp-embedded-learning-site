What Are Operators? The Tools of Programming

An operator is a symbol that performs an action on data. Think of it like buttons on a scientific calculator—each button does something specific. The + button adds, the × button multiplies, the ÷ button divides. Programming operators work exactly the same way, but they're much more powerful because they can work on any data type and in combinations.

Without operators, a program would just be a collection of storage boxes (variables) with no way to use them. Operators are how you:

  • Transform data — Convert sensor readings, scale analog values, calculate averages
  • Make decisions — Check if temperature exceeds a limit, if movement was detected, if safe conditions are met
  • Control hardware — Set individual hardware switches on/off, configure registers, communicate with peripherals

Why do we need so many operators? Different tasks require different tools. Each category of operators—arithmetic, comparison, logical, bitwise—solves specific problems that come up constantly in embedded systems.

Operators are the bridge between DATA (stored in variables) and ACTIONS (what your program actually does with the data).

Arithmetic Operators: Performing Calculations

Arithmetic operators do mathematical operations. You learned these in school, but in embedded systems you need to understand not just what they do, but why the details matter. Incorrect use of arithmetic operators is the source of many subtle bugs.

The Five Basic Operators

These are the building blocks you use to process sensor data, calculate values, and control behavior:

  • + (addition) — Combines two values: speed + offset = adjusted_speed
  • - (subtraction) — Finds differences: current_value - reference_value = deviation
  • * (multiplication) — Scales values: raw_adc * scaling_factor = millivolts
  • / (division) — Reduces values: total_millivolts / number_of_readings = average
  • % (modulo) — Finds the remainder: time_ms % 1000 = milliseconds_within_second

The modulo operator (%) is especially useful in embedded systems. It finds the remainder after division. For example, if you've counted 1,347 milliseconds and you want to know "how many milliseconds into the current second?", use 1347 % 1000 = 347. This is how you create repeating patterns and cyclical behavior.

Increment and Decrement: Quick Shortcuts

The operators ++ and -- are shortcuts for "add 1" and "subtract 1". You'll see these constantly in loops and counters. There are two forms, and the difference matters:

uint8_t counter = 5;

counter++;       // Post-increment: use 5 first, then change to 6
++counter;       // Pre-increment: change to 7 first, then use 7

counter--;       // Post-decrement: use 7 first, then change to 6
--counter;       // Pre-decrement: change to 5 first, then use 5

In embedded systems, you'll usually just see these in loops where the distinction doesn't matter:

for (uint8_t i = 0; i < 255; i++) {
    digitalWrite(LED_PIN, HIGH);
    delay(100);
    // i++ adds 1 at the end of each loop
}

Integer Division: The Hidden Trap

This is where a LOT of embedded systems bugs hide. When you divide two integers in C++, the result is truncated (rounded down) to an integer—the decimal part is discarded completely.

uint8_t result = 10 / 3;    // Result is 3 (not 3.333!)
// The decimal portion is lost forever

uint8_t average = (5 + 6 + 7) / 3;  // 18 / 3 = 6 (correct)
uint8_t average = (5 + 5 + 4) / 3;  // 14 / 3 = 4 (should be 4.67, loses precision)

Why this matters: Imagine converting raw sensor data to voltage. Your sensor reads 512 (out of 1024) and you want to know the voltage. If you divide first, you get 512 / 1024 = 0, then 0 * 5000 = 0V (wrong!). But if you multiply first, you get 512 * 5000 / 1024 = 2500mV = 2.5V (correct!).

// CORRECT - multiply first to preserve precision
uint16_t voltage_mv = (raw_adc * 5000) / 1024;

// WRONG - divides first, loses precision
uint16_t voltage_wrong = raw_adc / 1024 * 5000;  // Results in 0

Key rule: Always multiply before dividing to keep precision. The order of operations matters critically in integer math.

Order of Operations (PEMDAS)

Just like in mathematics, C++ follows a specific order: Multiplication and Division happen before Addition and Subtraction. Without parentheses, 10 + 5 * 2 equals 20 (multiply 5*2 first, then add 10). In embedded code, always use parentheses to make your intent crystal clear to yourself and other developers:

// Without parentheses: multiply happens first
uint8_t result1 = 10 + 5 * 2;   // = 10 + 10 = 20

// With parentheses: makes your intent explicit
uint8_t result2 = (10 + 5) * 2; // = 15 * 2 = 30

// Essential in sensor calculations
uint16_t scaled = (sensor_value * 100) / 255;  // Multiply first, divide second

Comparison Operators: Making Decisions

Comparison operators are how your program asks questions about data and makes decisions. They compare two values and answer with yes (true, or 1) or no (false, or 0). This is the foundation of all decision-making in code.

The Six Comparison Operators

Each one answers a specific question:

  • == (equal) — Does this value equal that value?
  • != (not equal) — Are these values different?
  • < (less than) — Is this value smaller than that one?
  • <= (less than or equal) — Is this value not larger than that one?
  • > (greater than) — Is this value larger than that one?
  • >= (greater than or equal) — Is this value not smaller than that one?

Each comparison returns either 1 (true) or 0 (false). You can store this result and use it:

uint8_t temperature = 25;
uint8_t is_hot = (temperature > 30);    // Evaluates to 0 (false)
uint8_t is_warm = (temperature >= 20);  // Evaluates to 1 (true)
⚠️ Most Common Mistake: = vs ==

Don't confuse assignment (=) with comparison (==). One sets a value, the other asks a question:

// WRONG - assigns 30 to temperature, always true
if (temperature = 30) {
    // This ALWAYS executes!
}

// CORRECT - asks if temperature equals 30
if (temperature == 30) {
    // Only executes if temperature actually is 30
}

Practical Example: Safety Monitoring

A thermostat system needs to check multiple boundaries. Temperature should stay between 20°C and 30°C:

#define LOWER_LIMIT 20
#define UPPER_LIMIT 30

uint8_t current_temp = 25;
uint8_t too_cold = (current_temp < LOWER_LIMIT);   // 0 (false)
uint8_t too_hot = (current_temp > UPPER_LIMIT);    // 0 (false)
uint8_t in_range = (current_temp >= LOWER_LIMIT && current_temp <= UPPER_LIMIT);  // 1 (true)

Logical Operators: Combining Conditions

Real embedded systems rarely make decisions based on a single condition. Usually you need to check multiple conditions at once: "Is temperature too high AND is the pump running?" or "Is the emergency button pressed OR is the timer expired?". Logical operators let you combine comparisons into complex decisions.

The Three Logical Operators

  • && (AND) — Both conditions must be true
  • || (OR) — At least one condition must be true
  • ! (NOT) — Reverses true to false, false to true

AND (&&): Both Conditions Required

Use AND when BOTH conditions must be true for something to happen. Think: "If the door is closed AND the key is inserted, start the car."

uint8_t door_closed = 1;  // true
uint8_t key_inserted = 1; // true

if (door_closed && key_inserted) {
    start_engine();  // Only runs if BOTH conditions are true
}

OR (||): At Least One Condition Required

Use OR when ANY condition being true should trigger an action. Think: "If the battery is low OR the temperature is high, trigger a warning."

uint16_t battery_mv = 2400;  // low
uint8_t temperature = 50;    // high

if (battery_mv < 2500 || temperature > 45) {
    activate_warning_led();  // Runs if EITHER condition is true
}

NOT (!): Reversing Logic

Use NOT to reverse a condition. "If the system is NOT in sleep mode, process sensor data."

uint8_t in_sleep_mode = 0;  // false (we're NOT in sleep mode)

if (!in_sleep_mode) {
    read_sensors();  // Runs because NOT(false) = true
}

Complex Logic: Combining Multiple Operators

Real systems combine many conditions. Always use parentheses to make the logic crystal clear:

// Safety system: sound alarm if (TEMPERATURE TOO HIGH AND PRESSURE TOO HIGH) OR (EMERGENCY BUTTON PRESSED)
uint8_t temp_high = (current_temp > 60);
uint8_t pressure_high = (current_pressure > 100);
uint8_t emergency = (button_state == PRESSED);

if ((temp_high && pressure_high) || emergency) {
    activate_alarm();  // Runs if either: (both temp and pressure high) OR (emergency button pressed)
}

Conditional Statements: Making Your Program Make Decisions

Operators like comparisons return true/false, but what do you DO with that result? Conditional statements are where you actually use operators to control program flow. They let you say: "If this condition is true, execute this code. Otherwise, execute that code." This is how embedded systems respond to sensor readings, button presses, and safety concerns.

The if Statement: Execute Code When True

The simplest conditional statement. It executes a block of code only if a condition is true:

if (condition) {
    // This code runs ONLY if condition is true
    // If condition is false, this block is skipped entirely
}

Real example:

uint8_t temperature = 35;

if (temperature > 30) {
    activate_cooling();  // Only runs if temperature exceeds 30
}

cout << "Check complete" << endl;  // This ALWAYS runs

The cooling is only activated if the condition is true. The final cout runs regardless because it's outside the if block.

The else Statement: Do Something When False

Execute alternative code when the condition is false. Think: "If it's raining, close windows; otherwise, leave them open."

if (condition) {
    // Runs if condition is true
} else {
    // Runs if condition is false
    // One of these two blocks ALWAYS executes
}

Real embedded example:

uint8_t button_state = 0;  // 0 = not pressed, 1 = pressed

if (button_state == 1) {
    activate_motor();
} else {
    deactivate_motor();
}

Critical insight: With if/else, exactly ONE block executes. Never both, never neither. Your program always takes one path.

The else if Statement: Multiple Conditional Paths

When you need to check multiple conditions in sequence. "If it's too cold, turn on heating. Else if it's too hot, turn on cooling. Otherwise, do nothing."

if (condition_1) {
    // Runs if condition_1 is true
} else if (condition_2) {
    // Runs if condition_1 is false AND condition_2 is true
} else if (condition_3) {
    // Runs if condition_1 and condition_2 are false AND condition_3 is true
} else {
    // Runs if all conditions above are false
}

This is how you implement multi-level safety systems. Real example:

#define TEMP_COLD 15
#define TEMP_NORMAL_LOW 20
#define TEMP_NORMAL_HIGH 25
#define TEMP_HOT 30

uint8_t current_temp = 22;

if (current_temp < TEMP_COLD) {
    // Extremely cold: emergency heating
    set_heater_mode(MAX);
    sound_alert();
} else if (current_temp < TEMP_NORMAL_LOW) {
    // Cool: gentle heating
    set_heater_mode(MEDIUM);
} else if (current_temp > TEMP_HOT) {
    // Hot: activate cooling
    set_cooler_mode(MAX);
    sound_alert();
} else if (current_temp > TEMP_NORMAL_HIGH) {
    // Warm: gentle cooling
    set_cooler_mode(LOW);
} else {
    // Normal: do nothing
    set_heater_mode(OFF);
    set_cooler_mode(OFF);
}

Key rule: The first condition that evaluates to true executes its block, then the entire if/else chain terminates. Remaining conditions are NOT checked. This prevents multiple blocks from running.

Combining Conditions with Logical Operators

Use logical operators (&&, ||, !) inside conditions for complex decision logic:

uint16_t voltage = 3200;
uint8_t temperature = 45;
uint8_t user_override = 0;

if ((voltage > 3000) && (temperature < 50) && !user_override) {
    // Start operation ONLY if: voltage adequate AND temp safe AND user hasn't overridden
    start_system();
}

if ((temperature > 50) || (voltage < 2500) || user_override) {
    // Emergency stop if: ANY of these is true (too hot OR voltage low OR user override)
    emergency_stop();
}

Common Pattern: Safety-Critical Decision

This is how real embedded systems protect themselves:

#define SAFE_VOLTAGE_MIN 4500  // mV
#define SAFE_VOLTAGE_MAX 5500  // mV
#define SAFE_TEMP_MAX 50       // °C
#define SAFE_PRESSURE_MAX 100  // PSI

uint16_t supply_voltage = 4800;
uint8_t system_temp = 45;
uint8_t system_pressure = 75;

// Check if ALL safety conditions are met
if ((supply_voltage >= SAFE_VOLTAGE_MIN) && 
    (supply_voltage <= SAFE_VOLTAGE_MAX) &&
    (system_temp < SAFE_TEMP_MAX) &&
    (system_pressure < SAFE_PRESSURE_MAX)) {
    
    operate_safely();  // Only runs if EVERY condition is true
} else {
    shutdown_immediately();  // If ANY condition fails, shutdown
}

Nested Conditionals: Making Decisions Within Decisions

You can put if statements inside other if statements (nesting). Use with care—too much nesting makes code hard to read:

if (user_pressed_button) {
    if (system_ready) {
        if (!error_exists) {
            start_system();
        } else {
            display_error();
        }
    } else {
        display_message("System not ready");
    }
}

This works, but it's getting hard to follow. Better approach using logical operators:

if (user_pressed_button && system_ready && !error_exists) {
    start_system();
} else if (user_pressed_button && !system_ready) {
    display_message("System not ready");
} else if (user_pressed_button && error_exists) {
    display_error();
}

Rule: Avoid excessive nesting. When you indent more than 2-3 levels, consider using logical operators to simplify the code.

Conditional statements are how your program makes decisions based on real-world data. Sensors provide input, conditionals analyze it, and your program acts accordingly. This is the foundation of all embedded systems.

Bitwise Operators: Controlling Hardware at the Bit Level

This is where embedded systems programming gets powerful. While most programmers think in bytes (0-255), embedded systems need to think in individual bits. A single byte has 8 bits, and each bit can control something different: 8 separate switches, 8 separate on/off states, or 8 separate hardware pins all in a single byte.

Bitwise operators let you manipulate individual bits without affecting others. This is how you actually control hardware registers, turn specific LED pins on/off, and read specific sensor states.

Understanding Bits and Binary

Before bitwise operators make sense, you need to understand binary. A byte contains 8 bits, each either 0 (off) or 1 (on):

uint8_t switches = 0b10110100;  // Binary notation
// Bit 7 = 1 (switch 7 is ON)
// Bit 6 = 0 (switch 6 is OFF)
// Bit 5 = 1 (switch 5 is ON)
// Bit 4 = 1 (switch 4 is ON)
// Bit 3 = 0 (switch 3 is OFF)
// Bit 2 = 1 (switch 2 is ON)
// Bit 1 = 0 (switch 1 is OFF)
// Bit 0 = 0 (switch 0 is OFF)

The Five Bitwise Operators

  • & (AND) — Keep bits that are 1 in both values
  • | (OR) — Set a bit to 1 if it's 1 in either value
  • ^ (XOR) — Flip a bit if it's different in the two values
  • ~ (NOT) — Flip all bits: 0→1, 1→0
  • << and >> (Shift) — Move bits left or right

Practical: Turning an LED On and Off

Most microcontrollers store LED states in a register—one byte with 8 bits, each bit controlling one LED. To turn LED 3 on (set bit 3 to 1):

#define LED_REGISTER  0x12      // Hardware register address
#define LED_3_BIT     (1 << 3)  // Bit 3 = 00001000

// Turn LED 3 ON by setting bit 3 to 1
LED_REGISTER |= LED_3_BIT;     // Set bit: OR with 1

// Turn LED 3 OFF by setting bit 3 to 0
LED_REGISTER &= ~LED_3_BIT;    // Clear bit: AND with NOT

// Check if LED 3 is on
if (LED_REGISTER & LED_3_BIT) {
    // LED 3 is on
}

Bitshift: Positioning Bits

The shift operators (<< and >>) move all bits left or right. Shifting left multiplies by 2; shifting right divides by 2. More importantly, they position bits:

uint8_t bit = 1;  // 00000001

bit << 0;  // 00000001 (bit 0)
bit << 1;  // 00000010 (bit 1)
bit << 2;  // 00000100 (bit 2)
bit << 3;  // 00001000 (bit 3)
bit << 4;  // 00010000 (bit 4)

// So to work with bit N, use: (1 << N)

Common Hardware Register Patterns

Here are the patterns you'll use constantly when working with hardware:

#define BIT_SET(byte, bit)      (byte |= (1 << bit))           // Set a bit to 1
#define BIT_CLEAR(byte, bit)    (byte &= ~(1 << bit))          // Set a bit to 0
#define BIT_TOGGLE(byte, bit)   (byte ^= (1 << bit))           // Flip a bit
#define BIT_CHECK(byte, bit)    ((byte & (1 << bit)) != 0)     // Check if bit is 1

// Usage:
uint8_t port = 0;
BIT_SET(port, 3);      // Set bit 3
BIT_CLEAR(port, 5);    // Clear bit 5
if (BIT_CHECK(port, 3)) {
    // Bit 3 is set
}

The Ternary Operator: Compact Decision Making

The ternary operator is a compact way to write simple if-else decisions in a single line. It's powerful for quick assignments but easy to misuse, so use it only when the code remains readable.

Syntax and How It Works

The ternary operator takes three parts (hence "ternary"):

variable = condition ? value_if_true : value_if_false;

Read it as: "If condition is true, use value_if_true; otherwise use value_if_false." Here's a practical example:

uint8_t temperature = 32;

// Simple if-else (verbose):
uint8_t fan_speed;
if (temperature > 30) {
    fan_speed = 255;  // Full speed
} else {
    fan_speed = 100;  // Low speed
}

// Same thing with ternary (compact):
uint8_t fan_speed = (temperature > 30) ? 255 : 100;

Both approaches do exactly the same thing, but the ternary version is one line. The ternary is evaluated left-to-right: evaluate the condition, then choose which value to use.

Real Embedded Systems Examples

Ternary operators are useful for status values, safety checks, and mode selection:

// System status based on voltage
uint16_t voltage = 4800;
uint8_t status = (voltage > 4500) ? 1 : 0;  // 1 = good, 0 = low

// LED indicator based on temperature
uint8_t temp_color = (current_temp > 40) ? RED : GREEN;

// Motor speed based on button state
uint8_t motor_pwm = button_pressed ? 200 : 0;

// Timeout decision
uint8_t timeout_reached = (elapsed_ms > MAX_TIMEOUT_MS) ? 1 : 0;

When NOT to Use Ternary

Avoid ternary operators when the decision is complex. Keep code readable:

// BAD - nested ternary is confusing!
uint8_t state = (temp > 50) ? 3 : (humidity > 80) ? 2 : 1;
// Hard to read and maintain

// GOOD - use if-else for complex logic
uint8_t state;
if (temp > 50) {
    state = 3;       // Overheating
} else if (humidity > 80) {
    state = 2;       // High humidity
} else {
    state = 1;       // Normal
}
Ternary operators are perfect for simple, one-line decisions. For anything more complex, use if-else statements. Code that's easy to read is code that's easy to debug.

Operator Precedence: What Gets Evaluated First?

When you write complex expressions with multiple operators, C++ needs to know which operations to perform first. This is called operator precedence. When in doubt, use parentheses to make it explicit:

uint8_t result = 2 + 3 * 4;     // 14, not 20 (multiply first)
uint8_t result = (2 + 3) * 4;   // 20 (parentheses force addition first)

// In conditional logic:
if (a && b || c && d) {    // Could be confusing
    // Both && operations happen before ||
}

// Better - explicit parentheses:
if ((a && b) || (c && d)) {  // Now it's crystal clear
    // Either (a AND b) OR (c AND d)
}
In embedded systems, clarity is more important than conciseness. Always use parentheses. Future-you will thank present-you.

Key Takeaways

  • Operators are the tools that let you DO things with data
  • Arithmetic operators calculate; always multiply before dividing to preserve precision
  • Comparison operators ask questions and return true/false
  • Logical operators combine conditions using AND, OR, NOT
  • Conditional statements (if/else/else if) execute code based on conditions
  • Bitwise operators control individual bits—the power of embedded systems
  • Ternary operator (? :) is perfect for simple one-line decisions
  • Always use parentheses to make operator order and logic explicit and clear

Project 1: Temperature Alert System

Objective: Create a system that monitors temperature and outputs warnings using comparison and logical operators.

Requirements:

  • Define temperature limits: normal (15-25°C), caution (25-35°C), critical (35+°C)
  • Store current temperature reading as a variable
  • Use comparison operators to check which range the temperature falls into
  • Output the alert level and specific temperature reading

Expected Output: "Temperature: 28C - CAUTION LEVEL"

#define NORMAL_HIGH 25
#define CAUTION_HIGH 35

uint8_t current_temp = 28;

cout << "Temperature: " << (int)current_temp << "C - ";

if (current_temp <= NORMAL_HIGH) {
    cout << "NORMAL" << endl;
} else if (current_temp <= CAUTION_HIGH) {
    cout << "CAUTION LEVEL" << endl;
} else {
    cout << "CRITICAL!" << endl;
}

// Display thresholds
cout << "Normal range: 15-" << (int)NORMAL_HIGH << "C" << endl;
cout << "Caution range: " << (int)(NORMAL_HIGH+1) << "-" << (int)CAUTION_HIGH << "C" << endl;

Key Learning: Use comparison operators (>, <, <=) with constants to create alert thresholds; output meaningful diagnostic information.

Project 2: Sensor Value Conversion

Objective: Convert raw ADC sensor readings to meaningful physical values using arithmetic operators correctly.

Requirements:

  • Simulate a sensor reading: 0-1023 (10-bit ADC)
  • Voltage reference: 0-5000 mV
  • Use CORRECT arithmetic: multiply before dividing to preserve precision
  • Convert voltage to meaningful unit (e.g., temperature or distance)
  • Display raw value, voltage, and converted value

Expected Output: "Raw: 512 | Voltage: 2500mV | Temperature: 25C"

#define ADC_RESOLUTION 1024     // 10-bit ADC
#define VREF_MV 5000            // 5V reference
#define SENSOR_PIN A0

uint16_t raw_adc = 512;

// CORRECT ORDER: Multiply first, then divide to preserve precision
uint16_t voltage_mv = (raw_adc * VREF_MV) / ADC_RESOLUTION;

// Convert voltage to temperature (LM35: 10mV per degree C)
uint8_t temperature = voltage_mv / 10;

cout << "Raw ADC: " << raw_adc << endl;
cout << "Voltage: " << voltage_mv << " mV" << endl;
cout << "Temperature: " << (int)temperature << " C" << endl;

Key Learning: Correct operator order (multiply before divide) is critical for precision; demonstrates real sensor data processing.

Project 3: Multi-Condition Safety Check

Objective: Implement a safety system using logical operators to combine multiple sensor conditions.

Requirements:

  • Monitor three conditions: temperature, pressure, and power supply
  • System is SAFE if: (temp < 50) AND (pressure < 100) AND (voltage > 4500mV)
  • System is UNSAFE if ANY condition fails
  • Output status and which conditions failed (if any)

Expected Output: "SAFE: All conditions normal" OR "UNSAFE: Temperature too high, Pressure OK, Voltage OK"

#define TEMP_LIMIT 50
#define PRESSURE_LIMIT 100
#define VOLTAGE_MIN 4500

uint8_t current_temp = 55;        // Too high!
uint8_t current_pressure = 75;    // OK
uint16_t supply_voltage = 4800;   // OK

uint8_t temp_ok = (current_temp < TEMP_LIMIT);
uint8_t pressure_ok = (current_pressure < PRESSURE_LIMIT);
uint8_t voltage_ok = (supply_voltage > VOLTAGE_MIN);

cout << "=== System Status ===" << endl;

if (temp_ok && pressure_ok && voltage_ok) {
    cout << "SAFE: All conditions normal" << endl;
} else {
    cout << "UNSAFE!" << endl;
    cout << "Temperature: " << (temp_ok ? "OK" : "FAIL") << endl;
    cout << "Pressure: " << (pressure_ok ? "OK" : "FAIL") << endl;
    cout << "Voltage: " << (voltage_ok ? "OK" : "FAIL") << endl;
}

Key Learning: Use logical operators (&&, ||) to combine conditions; implement real safety-critical logic for embedded systems.
