The Standard Library in Embedded Systems

C++ provides a massive standard library. But not all of it is appropriate for embedded systems. Some functions are designed for servers with gigabytes of RAM. Some rely on features not available in microcontrollers. Some cause runtime overhead that breaks real-time performance.

Our philosophy: Use the parts of the standard library that provide genuine value without hidden costs. Avoid dynamic memory, heavy abstractions, and features that increase binary size.

Embedded Standard Library Principle: Use only the headers you actually need. Each #include adds code to your binary. On a microcontroller with 32KB of flash, every kilobyte matters. Only include what you'll actually use.

The gold standard headers for embedded:

  • cstdio: printf, sprintf, fprintf—essential for debugging and formatted output
  • cstring: strlen, strcpy, strcmp—necessary for C-style strings and hardware communication
  • cmath: sin, cos, sqrt, pow—when you need mathematical calculations
  • cstdlib: malloc, free, abs, rand—general utilities (use malloc/free carefully!)
  • cassert: assert—catch programmer mistakes during development
  • cstdint: uint8_t, int16_t, uint32_t—fixed-width integer types (essential!)
  • climits: INT_MAX, INT_MIN, UINT_MAX—value limits

Headers to use carefully or avoid:

  • NO dynamic STL containers: std::vector, std::map, std::deque (use fixed arrays instead)
  • NO std::string: Use C-style char arrays or fixed-size buffers
  • NO exceptions: Many embedded systems disable exceptions (adds overhead, embedded systems crash anyway)
  • NO iostream: Use cstdio (printf) instead—larger binary, more overhead
  • NO threads: Use RTOS or interrupts instead

cstdio: Formatted Output with printf and sprintf

printf is your best friend in embedded systems. It's the foundation of debugging, logging, and communicating with the outside world.

Basic printf Usage

#include 

int main() {
    int count = 42;
    float voltage = 3.14159;
    
    printf("Count: %d\n", count);
    printf("Voltage: %.2f V\n", voltage);
}

Common format specifiers:

  • %d – decimal integer (int)
  • %u – unsigned integer
  • %x – hexadecimal (0x3F4A)
  • %f – floating point (3.14159)
  • %.2f – floating point with 2 decimals (3.14)
  • %c – single character
  • %s – null-terminated string

sprintf: Formatted String Building

sprintf writes formatted output to a string buffer instead of the console. This is essential when you need to construct messages for transmission:

char message[100];
int temperature = 25;

sprintf(message, "TEMP:%d,STATUS:OK\n", temperature);
// message now contains: "TEMP:25,STATUS:OK\n"

// Send to serial/network
send_message(message, strlen(message));
⚠️ Buffer Overflow Risk with sprintf

sprintf can write past your buffer if the formatted string is too long! Always use snprintf with a size limit:

char message[32];

// DANGEROUS - could overflow buffer
sprintf(message, "Value: %d", huge_number);

// SAFE - limits to 32 bytes
snprintf(message, sizeof(message), "Value: %d", huge_number);

Real Embedded Example: Sensor Logging

struct SensorReading {
    uint16_t temperature;
    uint16_t humidity;
    uint32_t timestamp;
};

void log_sensor(SensorReading *data) {
    char buffer[128];
    
    snprintf(buffer, sizeof(buffer),
             "[%lu] TEMP:%d HUMID:%d\n",
             data->timestamp,
             data->temperature,
             data->humidity);
    
    // Send to serial/SD card/network
    send_log_entry(buffer);
}

Key takeaway: printf and sprintf are lightweight, efficient, and available everywhere. Use them liberally for debugging and communication.

cstring: C-Style String Functions

Embedded systems use C-style strings (char arrays). The C++ std::string class adds overhead and dynamic memory. Learn these essential string functions:

Essential cstring Functions

#include 

// Length of string (counts until \0)
const char *msg = "Hello";
printf("Length: %zu\n", strlen(msg));  // Output: Length: 5

// Copy string
char buffer[50];
strcpy(buffer, "Source");  // buffer now contains "Source"

// Safe copy with length limit (preferred)
strncpy(buffer, "Source", sizeof(buffer) - 1);
buffer[sizeof(buffer) - 1] = '\0';  // Ensure null termination

// Compare strings
if (strcmp(buffer, "Source") == 0) {
    printf("Strings are equal\n");
}

// Case-insensitive comparison
if (strcasecmp(buffer, "source") == 0) {
    printf("Strings match (case-insensitive)\n");
}

// Find character in string
char *ptr = strchr("Hello World", 'o');  // Points to first 'o'

// Find substring
char *sub = strstr("Hello World", "World");  // Points to "World"
⚠️ Critical: strcpy and strcat are Dangerous

These functions can overflow buffers if the source is longer than the destination:

// DANGEROUS
char buffer[10];
strcpy(buffer, "This is a very long string");  // OVERFLOW!

// SAFE - use strncpy or better yet, strlcpy
char buffer[10];
strncpy(buffer, "This is a very long string", sizeof(buffer) - 1);
buffer[sizeof(buffer) - 1] = '\0';

Real Embedded Example: Parsing Commands

// Parse incoming serial command
void parse_command(char *received) {
    char cmd[32];
    int param;
    
    // Extract command (until space)
    if (sscanf(received, "%s %d", cmd, ¶m) == 2) {
        if (strcmp(cmd, "MOTOR") == 0) {
            set_motor_speed(param);
        } else if (strcmp(cmd, "LED") == 0) {
            set_led_brightness(param);
        }
    }
}
✓ Best Practice: Always Use Size-Limited Versions

Prefer strncpy, strncat, snprintf over strcpy, strcat, sprintf. Size limits prevent buffer overflows:

// Always do this:
strncpy(dest, src, sizeof(dest) - 1);
dest[sizeof(dest) - 1] = '\0';

// Or use strlcpy (BSD extension, very safe):
strlcpy(dest, src, sizeof(dest));

cmath: Mathematical Functions

When you need mathematical calculations, cmath provides trigonometric, exponential, and power functions. However, be aware that floating-point math is expensive on many microcontrollers.

Common cmath Functions

#include 

// Trigonometric
double angle = 0.5;  // radians
printf("sin(%.2f) = %.4f\n", angle, sin(angle));
printf("cos(%.2f) = %.4f\n", angle, cos(angle));
printf("tan(%.2f) = %.4f\n", angle, tan(angle));

// Exponential and logarithmic
printf("e^2 = %.4f\n", exp(2.0));
printf("ln(10) = %.4f\n", log(10.0));
printf("log10(100) = %.4f\n", log10(100.0));

// Power and roots
printf("2^8 = %.0f\n", pow(2.0, 8.0));
printf("sqrt(25) = %.1f\n", sqrt(25.0));

// Rounding
printf("ceil(3.2) = %.0f\n", ceil(3.2));  // 4.0
printf("floor(3.9) = %.0f\n", floor(3.9));  // 3.0
printf("round(3.5) = %.0f\n", round(3.5));  // 4.0

// Absolute value
printf("fabs(-5.5) = %.1f\n", fabs(-5.5));  // 5.5
⚠️ Floating-Point Performance Impact

Many microcontrollers lack hardware floating-point units. Using cmath functions means slow software emulation. Consider:

  • Using lookup tables for sin/cos instead of computing them
  • Integer math when precision allows
  • Fixed-point arithmetic for efficiency
// SLOW - floating point computation
float angle_rad = angle_deg * M_PI / 180.0;
float result = sin(angle_rad);

// FAST - lookup table (pre-computed)
const float sin_table[360] = {0.0, 0.01745, ...};
float result = sin_table[angle_deg];

Real Embedded Example: PID Controller

#include 

struct PIDController {
    float Kp, Ki, Kd;
    float integral;
    float last_error;
};

float compute_pid(PIDController *pid, float error) {
    // Proportional term
    float P = pid->Kp * error;
    
    // Integral term (prevent windup)
    pid->integral += error;
    if (pid->integral > 100.0) pid->integral = 100.0;
    if (pid->integral < -100.0) pid->integral = -100.0;
    float I = pid->Ki * pid->integral;
    
    // Derivative term
    float D = pid->Kd * (error - pid->last_error);
    pid->last_error = error;
    
    return P + I + D;
}

cstdlib: Utilities and Memory Management

cstdlib provides general utilities. Some are useful, some—like dynamic allocation—should be used carefully in embedded systems.

Useful Functions

#include 

// Absolute value
printf("abs(-42) = %d\n", abs(-42));  // 42
printf("labs(-1234567L) = %ld\n", labs(-1234567L));

// Random numbers
srand(time(NULL));  // Seed random generator
int random_value = rand() % 100;  // 0-99

// String to number conversion
int value = atoi("42");  // "42" -> 42
double dvalue = atof("3.14");  // "3.14" -> 3.14
long lvalue = strtol("0xFF", NULL, 16);  // "0xFF" -> 255 (hexadecimal)

// Malloc and free (use carefully!)
int *ptr = (int *)malloc(sizeof(int) * 100);
if (ptr) {
    ptr[0] = 42;
    free(ptr);
}

When to Use malloc/free

Dynamic allocation should be rare in production embedded systems. Use only when:

  • You allocate once at startup (not repeatedly)
  • You pre-calculate maximum needed size
  • You check for allocation failure (malloc returns NULL)
  • You actually need dynamic sizing
✗ Avoid: Repeated malloc/free 
// BAD - allocates/frees every loop
for (int i = 0; i < 1000; i++) {
    char *buffer = (char *)malloc(100);
    process(buffer);
    free(buffer);  // Fragmentation!
}
✓ Preferred: Allocate once, reuse 
// GOOD - allocate once, use repeatedly
char *buffer = (char *)malloc(100);
if (!buffer) {
    return ERROR;
}
for (int i = 0; i < 1000; i++) {
    process(buffer);
}
free(buffer);

cassert: Catch Programmer Mistakes

assert() is a development-time safety check. It forces you to think about invariants: "This should always be true. If it's not, something is very wrong."

Using assert

#include 

void set_temperature(int temp) {
    // Temperature should always be in valid range
    assert(temp >= -40 && temp <= 125);
    
    // This condition should always be true, but just to be safe
    assert(temp != UNDEFINED_VALUE);
    
    update_sensor_value(temp);
}

void process_array(int *arr, int size) {
    // Null check
    assert(arr != NULL);
    
    // Size should be valid
    assert(size > 0);
    assert(size < MAX_SIZE);
    
    for (int i = 0; i < size; i++) {
        process_element(arr[i]);
    }
}

What assert does:

  • In DEBUG builds: If condition is false, prints error message and halts
  • In RELEASE builds: assert can be disabled with `-DNDEBUG` compiler flag

static_assert: Compile-Time Checks

static_assert checks conditions at compile time, catching errors before code ever runs:

#include 

// Verify that uint16_t is exactly 2 bytes
static_assert(sizeof(uint16_t) == 2, "uint16_t is not 2 bytes!");

// Verify array size
uint8_t buffer[256];
static_assert(sizeof(buffer) < AVAILABLE_MEMORY, "Buffer too large!");

// Compile-time type checks
struct Device {
    uint8_t id;
    uint16_t serial;
};
static_assert(sizeof(Device) == 3, "Unexpected struct size!");
Production Reality: Use assertions during development to catch bugs early. Disable assertions in released firmware to save code space and remove any performance impact.

Standard Library Features to Avoid in Embedded Systems

NO: std::vector, std::map, std::deque

These STL containers use dynamic memory and are overkill for embedded systems:

// AVOID in embedded
std::vector data;  // Dynamic, can fragment memory
data.push_back(42);     // Allocates at runtime

// INSTEAD: Use fixed arrays
int data[MAX_SIZE];
int data_count = 0;
data[data_count++] = 42;  // Predictable, no allocation

NO: std::string

std::string handles memory for you but adds overhead:

// AVOID in embedded
std::string msg = "Hello";  // Dynamic allocation, overhead

// INSTEAD: Use C-style strings
char msg[] = "Hello";  // Fixed size, no overhead

NO: iostream (cout, cin)

iostream is much larger than cstdio, adds buffering overhead:

// AVOID: Larger binary, more overhead
#include 
std::cout << "Count: " << 42 << std::endl;

// INSTEAD: Lightweight, efficient
#include 
printf("Count: %d\n", 42);

NO: Exceptions (on resource-constrained platforms)

Exception handling adds code size. Many embedded systems disable it:

// AVOID: Uses exception overhead (if enabled)
try {
    risky_operation();
} catch (...) {
    handle_error();
}

// INSTEAD: Return error codes
int result = risky_operation();
if (result != SUCCESS) {
    handle_error();
}
⚠️ Binary Size Impact

Using STL containers and iostream can double your binary size! On a 32KB microcontroller, this is catastrophic. Use lightweight alternatives.

Key Takeaways

  • Use cstdio (printf, sprintf) for formatted output—efficient and available everywhere
  • Use cstring functions for character arrays (but always check sizes!)
  • Use cmath for mathematical functions (but consider lookup tables for performance)
  • Use cassert during development to catch mistakes; use static_assert for compile-time checks
  • Avoid STL containers (std::vector, std::map) in embedded systems
  • Avoid std::string; use C-style char arrays instead
  • Avoid iostream; use cstdio (printf) instead
  • Use malloc/free sparingly: allocate once at startup, not repeatedly
  • Every #include adds to binary size—only include what you need

Project 1: Serial Command Parser

Objective: Parse incoming serial commands and execute them.

Requirements:

  • Receive strings like "MOTOR:100" or "LED:255"
  • Parse command and parameter using sscanf or manual parsing
  • Execute appropriate action
  • Validate parameters are in valid range
  • Send formatted response back
#include 
#include 

void parse_command(char *received) {
    char cmd[32];
    int value;
    char response[64];
    
    // Parse: "COMMAND:VALUE"
    if (sscanf(received, "%[^:]:%d", cmd, &value) == 2) {
        // Validate value
        if (value < 0 || value > 255) {
            sprintf(response, "ERROR: Invalid value %d\n", value);
        }
        else if (strcmp(cmd, "MOTOR") == 0) {
            set_motor_speed(value);
            sprintf(response, "MOTOR set to %d\n", value);
        }
        else if (strcmp(cmd, "LED") == 0) {
            set_led_brightness(value);
            sprintf(response, "LED set to %d\n", value);
        }
        else {
            sprintf(response, "ERROR: Unknown command: %s\n", cmd);
        }
    } else {
        sprintf(response, "ERROR: Parse failed\n");
    }
    
    // Send response
    Serial.print(response);
}

Key Learning: sscanf parses formatted strings from input. Always validate parameters before use.

Project 3: Fixed-Point Math Alternative to Floating-Point

Objective: Implement PID controller using integer math instead of floats for efficiency.

Requirements:

  • Implement PID controller using integer math (shift instead of multiply by fraction)
  • Compare speed vs floating-point version
  • Verify results are close enough (acceptable error)
  • Measure code size difference
#include 

// Integer PID (FAST - uses shifts instead of multiply)
struct IntPID {
    int16_t Kp_scaled;  // Kp * 256 (scaled)
    int16_t Ki_scaled;
    int16_t Kd_scaled;
    int32_t integral;
    int16_t last_error;
};

int16_t compute_int_pid(IntPID *pid, int16_t error) {
    // Proportional: error * Kp / 256
    int32_t P = ((int32_t)error * pid->Kp_scaled) >> 8;
    
    // Integral: sum * Ki / 256
    pid->integral += error;
    if (pid->integral > 10000) pid->integral = 10000;
    if (pid->integral < -10000) pid->integral = -10000;
    int32_t I = (pid->integral * pid->Ki_scaled) >> 8;
    
    // Derivative: (error - last) * Kd / 256
    int32_t D = ((int32_t)(error - pid->last_error) * pid->Kd_scaled) >> 8;
    pid->last_error = error;
    
    int32_t output = P + I + D;
    // Clamp to int16_t range
    if (output > 32767) output = 32767;
    if (output < -32768) output = -32768;
    
    return (int16_t)output;
}

Key Learning: Bitwise shifts (>> 8 means divide by 256) are much faster than floating-point when you need precision.
